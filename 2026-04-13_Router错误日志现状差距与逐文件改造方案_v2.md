## 0. 结论先说

当前日志体系 **不完整**，且“渠道/支付/分组”三类错误的上下文字段覆盖明显不均衡：

1. 渠道错误：主干（`relay`）相对完整，但 adaptor 内部错误大量丢上下文。  
2. 支付错误：覆盖最弱，`callback` 关键失败分支几乎无日志。  
3. 分组/计费错误：有日志，但很多是 `WARN` 文本且不进 `error.log`，值班视角不连续。  
4. router 内部错误：有启动/中间件错误，但结构化字段缺失，无法直接喂飞书卡片。

## 1. `error.log` 机制现状（代码级）

### 1.1 现在是怎么写到 `logs/error.log` 的

- 初始化：`common/logger/logger.go:64-90`  
  - 创建 `router.log` 与 `error.log`，并设置 `gin.DefaultErrorWriter`。
- 路由：
  - `logHelper`（`common/logger/logger.go:223-248`）
  - `apiLogHelper`（`common/logger/logger.go:255-275`）
  - `relayLogHelper`（`common/logger/logger.go:278-298`）
- 规则：仅 `ERROR/FATAL` 会额外写入 `error.log`（`isErrorLogLevel` + `if isErrorLogLevel(level)`）。

### 1.2 边界与问题

1. `error.log` 当前是文本行，不是 JSON Lines，不利于飞书模板变量映射。  
2. `WARN` 不进 `error.log`，导致很多“有价值但未到 ERROR 的排障线索”断层。  
3. `logging.only_one_log_file` 已配置但未在 logger 使用（`common/config/config.go:196` + `common/runtime_config.go:502` 仅赋值）。

## 3. 回答你最关心的问题：哪些报错“现在就能加渠道/用户/endpoint”

可以，且大多数不需要改架构，只是改日志调用参数：

1. **adaptor 公共请求层** 已有 `c *gin.Context + meta *meta.Meta`：  
   - 直接可取 `meta.UserId/meta.Group/meta.ChannelId/meta.ActualModelName/meta.RequestURLPath`。  
   - 文件：`internal/relay/adaptor/common.go`。

2. **各 adaptor stream handler** 已有 `c *gin.Context`：  
   - 直接可取 `c.GetString(ctxkey.ChannelId/Id/Group/RequestModel/OriginalModel)`。  
   - 文件：`openai/tencent/gemini/anthropic/.../main.go`。

3. **计费扣减失败** 函数签名已带全量参数：  
   - `PostConsumeQuota(ctx, tokenId, quotaDelta, totalQuota, userId, groupID, channelId, ..., modelName, tokenName, ...)`。  
   - 文件：`internal/relay/billing/billing.go`。

4. **支付查单/回调** 业务对象里已有 `order.UserID/order.Id/provider_order_id`，控制器已有请求体：  
   - 文件：`internal/admin/model/topup_order_external_pay.go`、`internal/admin/controller/topup_callback.go`。

## 4. 具体“怎么改”方案（函数级，先方案）

## P0（先做，低风险高收益）

1. 补齐 relay 最终失败字段（不动业务逻辑）  
   - 文件：`internal/admin/controller/relay.go`  
   - 在 `FAIL` 日志增加：`endpoint=requestPath`、`user_id=userId`、`error_code=errorCodeString(bizErr.Code)`、`error_type=bizErr.Error.Type`。

2. 补齐 adaptor 公共请求日志上下文  
   - 文件：`internal/relay/adaptor/common.go`  
   - `UPSTREAM_ERR/UPSTREAM_RESP` 增加：`channel_id/user_id/group/model/endpoint`（从 `meta` + `c` 取）。

3. 把计费扣减失败从“无上下文 SysError”改为“带字段 Errorf/RelayErrorf”  
   - 文件：`internal/relay/billing/billing.go`、`internal/relay/controller/image.go`、`internal/relay/controller/video.go`。

4. 支付 callback 全链路补日志  
   - 文件：`internal/admin/controller/topup_callback.go`  
   - 在下列失败点都加日志：
   - `expectedToken==""`（配置错误）
   - token 校验失败
   - `ShouldBindJSON` 失败
   - `ApplyTopupOrderCallbackWithDB` 失败
   - `FulfillTopupOrderWithDB` 失败

## P1（第二批，提升支付可观测性）

1. 为 `queryTopupOrderByExternalPayAPI` 增加与 create 对称的失败日志函数  
   - 文件：`internal/admin/model/topup_order_external_pay.go`  
   - 新增 `logExternalPayQueryFailure(...)`，覆盖 HTTP 非 2xx、JSON 解析失败、`payloadResponse.Code != 1`。

2. 用户主动 refresh/cancel 失败落日志  
   - 文件：`internal/admin/controller/user/handler.go`（`RefreshTopUpOrder/CancelTopUpOrder`）。

## P2（结构化事件统一层）

1. 新增统一错误事件写入器（JSON Lines）  
   - 计划文件：`common/logger/error_event.go`。  
   - 将 `domain/subtype/error_code/impact/...` 固化为结构体字段。

2. 渐进替换 `openai.ErrorWrapper`  
   - 现状：`internal/relay/adaptor/openai/util.go:11-13` 使用 `context.TODO()`。  
   - 方案：保留旧函数兼容，新增 `ErrorWrapperWithContext(ctx, ...)`；先替换高频路径（text/image/audio/video + common adaptor）。

## 5. 建议新增但当前还缺失的排错字段（你问的“还有哪些有帮助”）

建议补这些字段，值班同学能直接看出“问题 + 影响”：

1. `domain/subtype`：`channel/payment/group_billing/router_internal`。  
2. `error_code`（内部统一码）+ `provider_code`（上游码）分离。  
3. `impact_scope`：`single_user/group/channel/system`。  
4. `impact_summary`：一句话影响描述（可直接进飞书卡片）。  
5. `endpoint`：规范化路径（如 `/v1/chat/completions`）。  
6. `billing_source`：`balance/package/package_fallback_balance`。  
7. `quota_delta/total_quota/pre_consumed_quota`：计费排障核心。  
8. `action_hint`：`retry/disable_channel/check_payment_callback/escalate`。  
9. `dedupe_key`：同类告警折叠。

## 6. 飞书落地对齐（只列最关键）

建议直接从结构化事件映射：

- `alert_title <- title`  
- `alert_level <- severity`  
- `alert_summary <- summary`  
- `error_code <- error_code`  
- `error_message <- error_message`  
- `user_id/group_id/channel_id/model_name/endpoint <- 同名字段`  
- `detail_url/runbook_url <- 同名字段`

这样你们后续换模板样式，不需要再改日志埋点。
