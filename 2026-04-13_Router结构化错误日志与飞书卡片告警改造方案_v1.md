# Router 结构化错误日志与飞书卡片告警改造方案 v1（仅方案，不改代码）

> 日期：2026-04-13  
> 状态：方案评审稿（Draft）  
> 约束：本稿只给改造方案，不包含实际代码修改。

---

## 1. 目标与范围

### 1.1 目标

把当前 `logs/error.log` 从“自由文本错误行”升级为“可机器消费的一行一事件 JSON”，并支持飞书卡片告警场景，让值班同学看到一条卡片就能回答：

1. 是什么问题（分类 + 错误码 + 业务结论）
2. 影响是什么（影响对象 + 影响范围 + 严重级别）
3. 该怎么处理（建议动作 + 详情链接 + runbook）

### 1.2 本期明确覆盖的三大类

1. 渠道错误（channel）
2. 支付错误（payment）
3. 分组/计费机制错误（group_billing）

并额外补充：

- 大模型调用错误（model_call，通常归到 channel 域下的细分子类）
- Router 自身错误（router_internal）

### 1.3 非目标

- 本期不重构全部历史日志文本
- 本期不替换 `router.log/api.log/relay.log` 的既有用途
- 本期不直接实现飞书交互回调（先打通“结构化日志 -> 卡片推送”）

## 2. 当前系统现状（基线）

### 2.1 文件日志现状

- `router.log`：系统主日志
- `api.log`：HTTP 访问日志
- `relay.log`：Relay 专项日志
- `error.log`：`ERROR/FATAL` 聚合（已生效）

### 2.2 核心问题

- `error.log` 目前是文本行，格式对人可读但对机器不友好
- 同类错误没有统一 `error_code/domain/subtype`，难以自动分流和值班统计
- 缺少“影响字段”与“建议动作字段”，告警消息只能“描述错误”，不能“指导处理”
- 飞书卡片模板变量需要稳定 key/value，当前文本日志无法直接映射

## 3. 结构化错误事件规范（v1）

采用 **JSON Lines**（每行一个 JSON 对象）写入 `logs/error.log`。

### 3.1 必填字段（MVP）

| 字段 | 类型 | 说明 |
|---|---|---|
| `event_type` | string | 事件类型，如 `channel_upstream_5xx` |
| `domain` | string | `channel / payment / group_billing / router_internal` |
| `subtype` | string | 子类，如 `upstream_quota`, `payment_callback_failed` |
| `severity` | string | `info / warning / error / critical` |
| `title` | string | 人类可读标题 |
| `summary` | string | 一句话摘要 |
| `biz_status` | string | 业务结论：`failed / degraded / recovered / ignored` |
| `error_code` | string | Router 内部统一错误码 |
| `error_message` | string | 面向值班的错误说明 |
| `occurred_at` | string | ISO8601 时间 |
| `service` | string | 服务名，如 `router` |
| `env` | string | `prod/staging/dev` |
| `trace_id` | string | 链路追踪 |
| `request_id` | string | 请求ID（可与 trace 同值） |
| `dedupe_key` | string | 告警去重键 |

### 3.2 影响与定位字段（强烈建议必填）

| 字段 | 类型 | 说明 |
|---|---|---|
| `impact_scope` | string | `single_user / group / channel / system` |
| `impact_summary` | string | 影响描述 |
| `user_id` | string | 受影响用户（可空） |
| `group_id` | string | 受影响分组（可空） |
| `channel_id` | string | 受影响渠道（可空） |
| `endpoint` | string | 受影响接口（如 `/v1/chat/completions`） |
| `model_name` | string | 模型名（可空） |

### 3.3 上游与上下文字段（可选但推荐）

| 字段 | 类型 | 说明 |
|---|---|---|
| `provider_status` | string | 上游原始状态 |
| `provider_code` | string | 上游返回码 |
| `http_status` | int | 对外 HTTP 状态 |
| `upstream_status` | int | 上游 HTTP 状态 |
| `retry_count` | int | 重试次数 |
| `root_cause` | string | 归因关键词 |
| `detail_url` | string | 内部详情页 |
| `runbook_url` | string | 处理手册 |
| `actions` | array | 动作建议（语义化） |
| `tags` | object | 扩展标签 |

### 3.4 示例（单行）

```json
{"event_type":"channel_upstream_quota_exhausted","domain":"channel","subtype":"upstream_quota","severity":"critical","title":"渠道上游额度不足","summary":"请求失败并触发降级","biz_status":"failed","error_code":"CHANNEL_UPSTREAM_QUOTA_EXHAUSTED","error_message":"上游返回额度不足，当前分组请求被拒绝","occurred_at":"2026-04-13T18:30:00+08:00","service":"router","env":"prod","trace_id":"202604131830001234","request_id":"202604131830001234","dedupe_key":"channel_upstream_quota_exhausted:group_x:channel_y","impact_scope":"group","impact_summary":"分组 group_x 下调用该渠道的请求将持续失败","user_id":"","group_id":"group_x","channel_id":"channel_y","endpoint":"/v1/chat/completions","model_name":"claude-sonnet-4-5","provider_status":"insufficient_quota","provider_code":"billing_hard_limit_reached","http_status":503,"upstream_status":403,"retry_count":1,"detail_url":"https://internal/router/ops/incidents/abc","runbook_url":"https://internal/router/runbook/channel-quota","actions":[{"id":"detail","label":"查看详情","mode":"link","url":"https://internal/router/ops/incidents/abc"}],"tags":{"team":"oncall","source":"relay"}}
```

## 4. 分类模型（值班视角）

### 4.1 域与子类（第一版）

1. `channel`
- `upstream_4xx`
- `upstream_5xx`
- `upstream_quota`
- `model_not_supported`
- `channel_disabled`
- `channel_retry_exhausted`

2. `payment`
- `topup_create_failed`
- `topup_query_failed`
- `payment_callback_failed`
- `payment_signature_invalid`
- `fx_sync_failed`

3. `group_billing`
- `group_daily_quota_exceeded`
- `package_reservation_settle_failed`
- `billing_currency_invalid`
- `quota_counter_inconsistent`

4. `router_internal`
- `db_unavailable`
- `redis_unavailable`
- `panic_recovered`
- `config_invalid`

### 4.2 每条错误都要有“影响判断”

最少包含：

- 影响对象：用户 / 分组 / 渠道 / 全局
- 是否可自动恢复：`recoverable=true/false`
- 值班动作建议：`observe / retry / disable_channel / rollback / escalate`

## 5. 与飞书卡片对接设计

## 5.1 映射原则

- `error.log` 只存结构化事件（事实层）
- 飞书模板渲染层做 `template_id + template_variable` 映射（展示层）
- 不在日志层拼死 UI 文案，避免被模板结构绑死

### 5.2 变量命名策略

- 全小写，下划线命名
- 变量名稳定，不随 UI 变化
- `template_variable` 从事件字段直接投影

### 5.3 按机器人类型降级

1. 自定义机器人：
- 仅静态卡片 + 链接按钮
- `actions.mode=callback` 自动降级为 `link`

2. 应用机器人：
- 可启用交互回调
- 回调处理接口需 3 秒内响应（先 ACK 后异步处理）

### 5.4 消息大小与截断

- 目标：卡片 payload < 30KB
- `error_message` 超长时截断 + `detail_url` 补全
- 堆栈信息不放卡片正文，落详情页

## 6. 对当前 Router 代码的改造计划（仅计划）

> 以下是“将来要改哪些位置”的路线图，不代表本次已经改动。

### 6.1 日志基础层

1. `common/logger/logger.go`
- 新增结构化错误写入入口（例如 `StructuredError(event ErrorEvent)`）
- 保留现有文本日志，先走“双写”策略：
  - 文本照旧写 `router.log`
  - 结构化错误写 `error.log`（JSON Lines）
- 增加字段脱敏（token、authorization、secret）

2. 新增 `common/logger/error_event.go`（计划）
- 定义 `ErrorEvent` 结构体
- 内置 `Validate()`、`Normalize()`、`ToJSONLine()`

### 6.2 业务映射层

1. `internal/admin/controller/relay.go`
- `FAIL/UPSTREAM_ERR/RETRY` 分支映射到 `domain=channel`
- 補齐 `impact_scope/group_id/channel_id/model_name/endpoint`

2. `internal/relay/adaptor/common.go` + `internal/relay/controller/error.go`
- 上游状态与 provider code 统一映射
- 规范 `provider_status/provider_code/root_cause`

3. 支付链路（topup / callback / FX）
- 覆盖 `domain=payment` 事件
- 明确 `order_id`, `provider_status`, `biz_status`

4. 分组/计费链路（quota reservation, settle）
- 覆盖 `domain=group_billing`
- 标注 `group_id` 和影响范围

### 6.3 告警发送层（建议独立）

新增 `internal/alert/feishu_sender`（计划）：

- 输入：结构化错误事件
- 输出：飞书消息（模板卡片优先）
- 策略：
  - 去重（`dedupe_key`）
  - 抑制（短窗同类聚合）
  - 升级（warning -> critical）

## 7. 分阶段实施计划

## Phase 0：规范冻结（1~2 天）

- 冻结 `ErrorEvent v1` 字段与分类字典
- 冻结飞书模板变量 key
- 产出事件字典文档与样例

**验收**：评审通过，字段不再随意变更。

## Phase 1：双写落地（2~3 天）

- 保持当前文本日志能力不变
- 同步输出结构化 `error.log` JSON Lines
- 做基础脱敏与字段校验

**验收**：error 事件 100% 为合法 JSON 行。

## Phase 2：高价值链路接入（3~5 天）

- 先接 relay/channel、payment、group_billing 三大域
- 每个域至少覆盖 3~5 个高频错误子类

**验收**：
- 覆盖率 >= 80%（以近 7 天 error 样本计）
- 每条事件都带 `domain/subtype/error_code/impact_scope`。

## Phase 3：飞书卡片投递（2~3 天）

- 打通模板卡片发送
- 支持自定义机器人降级策略
- 控制消息体大小与去重

**验收**：
- 值班群可稳定收到卡片
- 同类告警不会刷屏
- 卡片能直达详情/手册。

## Phase 4：观测与收敛（持续）

- 建立按 `domain/subtype` 的周报
- 下线无用字段，补全高缺失字段
- 根据值班反馈优化 `summary/impact/action` 文案

## 8. 质量门槛（Definition of Done）

1. 结构化完整性
- `error.log` 每行 JSON 解析成功率 100%
- 必填字段缺失率 < 0.1%

2. 告警可用性
- 值班同学看到卡片 30 秒内能判断：问题类型 + 影响范围 + 下一步动作

3. 稳定性
- 发送失败不影响主请求链路
- 告警链路故障可降级为仅落日志

4. 安全
- 无密钥/令牌/隐私字段泄漏

---

## 9. 风险与应对

1. 风险：历史调用点太多，接入成本高
- 应对：先覆盖高频高损失链路，分阶段推进

2. 风险：字段定义频繁变化导致模板失配
- 应对：字段版本化（`spec_version=v1`），模板按版本渲染

3. 风险：告警噪声过大
- 应对：`dedupe_key` + 窗口聚合 + severity 门限

4. 风险：卡片过长被飞书拒绝
- 应对：字段截断、堆栈外链、payload 长度预检

## 11. 一句话执行策略

先双写、再覆盖、后卡片；先保证“机器可读 + 值班可判”，再追求“交互闭环”。
