### 2.4 SSH 转发关键参数（决定是否可直接公网暴露）

`sshd -T` 关键值：

- `allowtcpforwarding yes`
- `gatewayports no`
- `permitopen any`

关键结论：

1. 反向隧道 `-R` 可用。
2. 但由于 `gatewayports no`，即使请求 `-R 0.0.0.0:PORT:...`，远端也会强制绑到 `127.0.0.1`（仅 VPS 本地可见，不直接对公网开放）。

### 2.5 现有公网入口结构

- `443` 由 `sslh` 接入：
  - SSH 流量 -> `127.0.0.1:22`
  - TLS 流量 -> `127.0.0.1:8443`（nginx）
- nginx 已有站点/反代。

这意味着：最稳妥做法是“隧道到 VPS 回环 + nginx 对外发布”，而不是直接改 SSH 网关转发策略。

### 2.6 临时可行性验证（已完成）

已做临时隧道（验证后已关闭）：

- 本机执行：`-R 127.0.0.1:17788:127.0.0.1:7788`
- VPS 上验证：`curl -I http://127.0.0.1:17788` -> `HTTP/1.1 200 OK`

同时验证：请求 `-R 0.0.0.0:17789:...` 后，VPS 实际监听仍是 `127.0.0.1:17789`（符合 `gatewayports no`）。

## 3. 方案对比（先给结论）

### 方案 A（推荐）：反向隧道 + Nginx 对外发布（不改 sshd）

路径：

1. 本机持续建立反向隧道到 VPS 回环端口（如 `127.0.0.1:17788`）。
2. VPS nginx 在现有 `443->8443` 入口上反代到 `127.0.0.1:17788`。

优点：

- 不改 `sshd` 全局安全策略。
- 复用现有 443 入口，用户侧访问体验好。
- 可在 nginx 层加鉴权、IP 白名单、限流。

缺点：

- 需要改 nginx 配置（可控、低风险）。

### 方案 B（可做但不推荐）：改 sshd `GatewayPorts clientspecified` + 直接 `-R 0.0.0.0`

优点：配置简单，直接公网端口映射。

缺点：

- 需要修改并重载 sshd。
- 风险面更大（任意反向端口更容易误暴露）。
- 仍要开 UFW 新端口。

### 方案 C：VPS 高位端口直出（socat/haproxy）

优点：不改 sshd。

缺点：

- 需要额外守护进程。
- 通常是裸端口，鉴权能力弱于 nginx HTTPS 入口。

## 4. 推荐实施蓝图（方案 A）

### 4.1 数据流

公网用户 -> `https://<你的域名>` (443) -> `sslh` -> nginx:8443 -> `127.0.0.1:17788` (VPS loopback tunnel) -> SSH reverse tunnel -> 本机 `127.0.0.1:7788`

### 4.2 执行步骤（拟定，未执行）

1. 本机隧道守护化（`systemd --user` 或 `autossh`）
2. VPS nginx 增加一个独立 `server_name`（建议专用子域名，不建议与现有站点路径混用）
3. nginx 里加基础防护：
   - `auth_basic`（至少先挡住默认 `admin/admin` 风险）
   - 可选 IP allowlist
   - WebSocket 头与较长超时
4. 验证链路：
   - 本机 `7788` -> VPS `127.0.0.1:17788` -> 公网域名
5. 最后再决定是否开放 IPv6 公网入口（当前从本机未完成 IPv6 可达性确认）

## 5. 风险与注意事项

- CodexHub 当前默认登录是 `admin/admin`，直接公网暴露风险高；建议先在前置层加认证，或先改应用默认口令。
- VPS 现有业务端口较多，避免端口冲突，尽量走域名 + 443 统一入口。
- `git.shengnw.win` 当前外测返回 502，说明现有站点链路里有待修复项；新入口建议独立域名，避免耦合。
