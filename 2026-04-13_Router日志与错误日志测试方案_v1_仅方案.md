## 2. 测试目标（你关心的“日志报错信息正常”）

定义“正常”= 同时满足以下 5 个维度：

1. 可写入：错误发生时确实有日志输出（不丢日志）
2. 级别正确：`INFO/WARN/ERROR/FATAL` 路由正确，`ERROR/FATAL` 进入 `error.log`
3. 格式正确：文本日志格式稳定；卡片日志是可解析 JSON 且字段完整
4. 上下文正确：trace_id、user/group/channel/model 等关键上下文字段可用
5. 语义正确：错误码、业务结论、影响范围与场景匹配

---

## 3. 推荐测试架构（分层）

## L1. Logger 基础契约测试（单元）

目标：验证 `common/logger` 本身不会写错文件、写错级别。

建议新增：

1. `common/logger/logger_test.go`
2. `common/logger/error_card_test.go`

关键用例：

1. `Info` 仅写 `router.log`，不写 `error.log`
2. `Error / Errorf / SysError / RelayErrorf / ApiLogf(ERROR)` 会写 `error.log`
3. `RelayWarnf` 仅写 `relay.log`（不进 `error.log`）
4. 含 `trace_id` 的 context 能在日志行中出现
5. 文本日志行符合固定格式（时间、级别、文件:行号、函数名、消息）
6. `EmitFeishuCardError` 输出单行 JSON，可反序列化
7. `normalizeErrorCardEvent` 默认值、长度截断、tag 排序、模板颜色映射正确
8. 并发写卡片（多 goroutine）不串行破坏（每行都是完整 JSON）
9. `FatalLog` 子进程测试（验证进程退出行为）

## L2. 业务告警卡片契约测试（包内单元）

目标：覆盖 8 个 `EmitFeishuCardError` 发射点的字段完整性。

建议新增测试文件：

1. `internal/admin/controller/relay_error_card_test.go`
2. `internal/transport/http/middleware/distributor_error_card_test.go`
3. `internal/relay/controller/group_quota_error_card_test.go`
4. `internal/relay/controller/image_error_card_test.go`
5. `internal/relay/controller/video_error_card_test.go`
6. `internal/relay/billing/billing_error_card_test.go`
7. `internal/admin/controller/topup_callback_error_card_test.go`
8. `internal/admin/model/topup_order_external_pay_error_card_test.go`

每个发射点统一断言：

1. `event_type/domain/subtype/severity` 非空且取值合法
2. `error_code/error_message/biz_status/impact_scope/impact_summary` 非空
3. 场景字段（user/group/channel/model/endpoint/http_status）按预期出现
4. `error.log` 中该行 JSON 的 `event` 与业务参数一致

## L3. HTTP/Relay 集成测试（端到端日志行为）

目标：验证“请求 -> 失败 -> 响应 + 日志”全链路。

建议新增：

1. `internal/transport/http/middleware/recover_test.go`
2. `internal/transport/http/middleware/relaylogger_test.go`
3. `internal/admin/controller/relay_logging_test.go`
4. `internal/admin/controller/topup_callback_logging_test.go`

关键场景：

1. panic 场景：返回 500，日志有 panic + stack + request body
2. relay 失败场景：`FAIL` + `END` 错误日志入 `relay.log` 与 `error.log`
3. 429/5xx 上游场景：重试日志 (`RETRY`) 与最终 `FAIL` 语义正确
4. 无可用渠道场景：请求被拒绝 + 发卡片 + `abortWithMessage` 记录 error
5. 回调 token 错误/JSON 错误/发货失败场景：HTTP 响应与卡片日志匹配

## L4. 全仓静态覆盖测试（保证“所有错误日志点”被追踪）

目标：避免未来新增错误日志点却没有测试或规范校验。

建议新增：

1. `internal/quality/logging_callsite_test.go`
2. `internal/quality/testdata/error_log_callsites.golden.json`

方法：

1. AST 或 `go list + parser` 扫描全仓 `logger.Error* / SysError* / Fatal* / Warn* / EmitFeishuCardError`
2. 与 golden 清单比对
3. 新增调用点时强制更新清单并补测试映射说明

收益：实现“覆盖所有错误日志调用点”的可持续机制，而不是一次性手工盘点。

## 4. 建议测试矩阵（按域）

| 域 | 代表文件 | 代表错误 | 重点断言 |
|---|---|---|---|
| channel | `internal/admin/controller/relay.go` | upstream 4xx/5xx/quota | FAIL 字段齐全；卡片 domain=channel |
| channel | `internal/transport/http/middleware/distributor.go` | no available channel | 拒绝响应 + critical 卡片 + endpoint/model |
| payment | `internal/admin/controller/topup_callback.go` | token invalid/payload invalid/apply failed/fulfill failed | `error_code` 与 `impact_summary` 正确 |
| payment | `internal/admin/model/topup_order_external_pay.go` | create/query 失败 | warn 文本 + payment 卡片 JSON 可解析 |

| group_billing | `internal/relay/billing/billing.go` | post-consume/cache update 失败 | card tags 含 token_id/quota_delta/total_quota |

| router_internal | `common/runtime_config.go` / `internal/app/app.go` | 配置/启动 fatal | 级别与文件路由正确 |

## 6. CI 建议（最小可行）

当前 `.github/workflows/compile_before_merge.yml` 只做 build。建议增加：

1. `go test ./common/logger ./internal/transport/http/middleware ./internal/relay/... ./internal/admin/...`
2. 后续再扩展到 `go test ./...`

说明：先从日志相关包开始，降低一次性改动风险。

---

## 7. 验收标准（可量化）

1. `common/logger` 与 `error_card` 测试通过率 100%
2. 8/8 卡片发射点均有对应测试
3. 关键失败场景（relay、callback、panic）均验证“响应 + 日志 + 错误卡片”三件套
4. 静态 callsite 清单与仓库一致（新增点必须评审）
5. CI 中日志测试任务默认执行

---

## 8. 风险与注意

1. `logger` 使用全局 `sync.Once`，测试需要注意初始化顺序（建议每个包首次日志前设置测试 logDir）
2. `FatalLog` 需子进程测试，不能在当前测试进程直接调用
3. 并发日志测试要避免依赖行顺序，应按“行数 + 可解析性 + 字段存在”断言
4. 部分历史 `Warn` 虽然语义上是错误风险，但按当前实现不会进 `error.log`，测试需按现有行为断言
